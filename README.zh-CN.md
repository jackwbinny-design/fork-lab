# Fork Lab

[English README](./README.md)

Fork Lab 是一个面向 GitHub fork 收藏的学习决策看板。

它想做的一件事很直接：

把一堆“先 fork 下来以后再看”的仓库，变成一个每周真的能推进的决策系统。

如果你的 GitHub 账号已经变成了“收藏很多、下一步不清楚”，那这个项目就是为这种状态设计的。

## 一眼看懂它在解决什么

Fork Lab 针对的是这样一种很具体的状态：

- fork 的速度远快于真正研究的速度
- 记得当时觉得它有意思，但已经忘了为什么
- 相关项目本来应该一起比较，却散落在列表里
- 需要的是每周几个优先项，而不是继续堆收藏

一句话概括：

> Fork Lab 想把 GitHub forks 从“保存过的链接”变成“结构化的学习队列”。

它解决的不是“怎么发现更多项目”，而是另一个更常见、也更麻烦的问题：

很多人会在 GitHub 上 fork 大量感兴趣的仓库，想着以后再研究。但时间一长，这些 fork 会慢慢变成一堆没有处理过的收藏：

- 项目很多
- 当时为什么 fork 已经忘了
- 哪些还值得看不清楚
- 每周到底该先研究哪几个也没有优先级

Fork Lab 就是针对这个问题做的。

它想把“被动收藏”变成“主动决策”：每周打开一次，看清哪些项目值得继续投入，哪些项目其实属于同一条学习路线，哪些应该留下做参考，哪些应该干脆丢掉。

## 打开以后你会看到什么

Fork Lab 不是一个单纯的仓库列表页，它主要围绕三层界面来组织：

- 一个会直接告诉你“这周先看什么”的 dashboard
- 一个用来理解“哪些项目本来就该放在一起看”的 cluster 层
- 一个用来搜索、筛选、判断单个项目的 library 视图

它想提供的体验是：

1. 打开首页。
2. 立刻看到这周重点。
3. 进入一个项目簇。
4. 真正跑一次测试。
5. 留下继续 / 保留 / 丢弃 的明确判断。

## 为什么会做这个项目

这个项目最初来自一个很具体的使用场景：

一个 GitHub 账号会不断 fork 各种感兴趣的开源项目，尤其是 AI agent、设计工具、自动化流程、浏览器增强、本地部署、知识管理这几类。fork 的时候往往是因为“先留着，之后看”，但几周或几个月之后，账号里已经堆了几百个 fork，真正被认真处理的却很少。

这时候，问题已经不是“发现项目”了，而是“如何判断和取舍”：

- 我当时为什么 fork 这个？
- 这个项目现在还活着吗？
- 它到底适不适合我继续投入？
- 哪些仓库其实是同一条能力路线，应该一起看？
- 这周最值得推进的 3 组是什么？

Fork Lab 想做的，就是给这些问题一个更清晰、更可执行的界面。

## Fork Lab 是什么

Fork Lab 是一个用 React + Vite 做的前端 Dashboard，它会把 GitHub fork 列表整理成一层“决策界面”。

它整合的不是单一数据，而是几层信息：

- 仓库基础信息
- 上游项目线索
- README 摘要与重点
- 人工整理的项目关系簇
- 本地保存的每周判断与行动记录

所以它不是一个普通的仓库列表页，而更像一个轻量的个人研究工作台。

## 从“产品角度”看它在做什么

Fork Lab 解决的并不是项目发现。

因为“发现项目”本身已经有很多渠道了：

- GitHub Trending
- 社交平台时间线
- newsletter
- 浏览器书签
- GitHub stars
- 朋友发来的链接

真正难的是发现之后。

一旦你已经收集过量，就会需要一套机制来处理：

- 上下文
- 排序
- 分组
- 每周复盘
- 明确决策

这才是 Fork Lab 作为产品的核心角度。

## 核心思路

Fork Lab 把 fork 收藏看成一个三段式流程：

1. 先把感兴趣的项目 fork 下来，避免链接丢失。
2. 再补充足够的上下文，帮助自己以后更快判断。
3. 最后按周复盘，让这些收藏真正变成行动，而不是继续堆积。

因此它的界面重点不是“展示全部仓库”，而是围绕这几个动作来组织：

- 精选候选
- 本周行动组
- 相关项目簇
- 留存 / 深挖 / 丢弃 决策

## 它是怎么工作的

### 1. 先把 fork 变成结构化库存

Dashboard 会从一个 GitHub 账号的 fork 仓库出发，把它们整理成可筛选、可排序的数据集。

每条记录可以包含：

- 仓库名和上游来源
- 分类和评分
- 项目描述与语言
- README 重点
- 起步动作建议
- 面向学习和判断的中文说明

### 2. 再按“项目簇”来理解，而不是单看单个仓库

很多仓库不适合孤立评估。

一个仓库可能是主项目，另外几个则可能是：

- 配套工具
- 更轻量的替代品
- 实现参考
- 数据集或提示词资料

Fork Lab 会把这些项目整理成 cluster，让你按主题复盘，而不是被随机的单个仓库打散注意力。

### 3. 最后收束到每周行动

它不会要求你把所有 fork 一次看完。

它真正想推动的是：

- 先挑少数几个强候选
- 跑一次真实测试
- 记下学到了什么
- 然后决定是继续、保留、转成产品方向，还是直接丢弃

## 适合谁

Fork Lab 比较适合这类人：

- 经常 fork 项目做研究或收集灵感的人
- 在 AI、设计、自动化、工具、副业探索这些方向来回切换的人
- 觉得浏览器书签、GitHub stars、原始 fork 列表都不够用的人
- 想把开源发现流程做成“每周可执行复盘”的人

它不是一个通用型 GitHub 数据分析平台。

更准确地说，它是一个个人学习与项目判断看板。

## 典型使用场景

- 一个每周都在 fork 新 agent 框架的 AI 工具研究者，需要更好的项目分流机制。
- 一个收集设计系统、组件库、产品参考仓库的设计师或独立开发者。
- 一个同时在看多种相邻开源工具，希望按“项目簇”而不是单个链接来判断的人。
- 一个想把 GitHub 上的开源发现，真正变成每周固定习惯的人。

## 当前功能

- React + Vite 单页 Dashboard
- 仓库总览与精选候选
- 本周行动组
- 相关项目 cluster 视图
- 本地决策状态与每周记录持久化
- 上游详情富化脚本

## 本地运行

```bash
npm install
npm run dev
```

默认开发地址：

```text
http://127.0.0.1:5173
```

## 构建

```bash
npm run build
```

## 数据刷新

```bash
npm run update:data
npm run enrich:details
npm run enrich:readmes
npm run refine:categories
```

一键刷新并重建：

```bash
npm run update:dashboard
```

## 部署到 GitHub Pages

仓库内已包含 GitHub Actions 工作流：

- 推送到 `main` 后自动构建
- 自动发布 `dist/` 到 GitHub Pages

首次启用时，请在 GitHub 仓库设置里确认：

1. `Settings -> Pages`
2. `Build and deployment` 选择 `GitHub Actions`

## 技术栈

- React 18
- Vite 8
- Lucide React
- Node.js 脚本做 GitHub 数据整理

它当前只专注解决一个核心问题：把大量 GitHub fork 收藏，从“看过但没处理”变成“能复盘、能理解、能行动”的项目池。
